# Symphony Python 实现进度

## 已完成的修改

### 1. LLM 供应商无关设计 ✅

**移除 Codex app-server 依赖：**
- 删除了所有 `codex` 相关的配置
- 删除了 `CodexConfig` 类
- 更新了 `SessionState` 使用 `llm_usage` 替代 `codex_*`
- 更新了 `OrchestratorState` 使用 `llm_totals` 替代 `codex_totals`

**新增 LLM 配置：**
- `LLMConfig` 类支持多种供应商
- 自动从环境变量读取配置
- 支持 `OPENAI_API_KEY`, `OPENAI_BASE_URL`, `OPENAI_MODEL`
- 支持 `ANTHROPIC_*`, `DEEPSEEK_*`, `GEMINI_*`, `AZURE_*`

### 2. 环境变量和 .env 文件支持 ✅

**配置优先级：**
1. WORKFLOW.md 显式设置
2. 环境变量
3. .env 文件
4. 默认值

**自动加载 .env 文件：**
- 工作流文件所在目录的 `.env`
- 当前工作目录的 `.env`
- 用户主目录的 `~/.symphony.env`

**新增文件：**
- `.env.example` - 环境变量示例
- `src/symphony/llm/` - LLM 客户端模块

### 3. LLM 客户端模块 ✅

**文件结构：**
```
src/symphony/llm/
├── __init__.py
├── client.py      # 主要 LLM 客户端
└── providers.py   # 供应商配置
```

**支持的供应商：**
- OpenAI (默认)
- Anthropic Claude
- DeepSeek
- Google Gemini
- Azure OpenAI

**特性：**
- 统一的 `Message` 和 `LLMResponse` 接口
- 同步和流式响应支持
- 自动重试机制
- 供应商特定的消息格式转换

### 4. Agent 实现 ✅

**文件结构：**
```
src/symphony/agents/
```

## 配置文件示例

### .env 文件
```bash
# LLM (OpenAI)
OPENAI_API_KEY=sk-...
OPENAI_BASE_URL=https://api.openai.com/v1
OPENAI_MODEL=gpt-4

# Linear
LINEAR_API_KEY=lin_api_...
```

### WORKFLOW.md
```yaml
---
llm:
  provider: openai
  temperature: 0.7
  max_tokens: 4096
  # api_key, base_url, model 从环境变量读取

tracker:
  kind: linear
  project_slug: "your-project"
  api_key: $LINEAR_API_KEY
---
```

## 与原设计的差异

| 原设计 | 新设计 |
|--------|--------|
| Codex app-server 依赖 | LLM 供应商无关 |
| `codex.command` | `llm.provider` |
| `codex.approval_policy` | 移除 |
| `codex.thread_sandbox` | 移除 |
| `codex.stall_timeout_ms` | `agent.stall_timeout_seconds` |
| 仅支持 Codex | 支持 OpenAI, Anthropic, DeepSeek, Gemini, Azure |
| AgentRunner 使用 Codex | SymphonyAgent 使用 LLMClient |

## 使用方法

```bash
# 1. 设置环境变量
export OPENAI_API_KEY="sk-..."
export LINEAR_API_KEY="lin_api_..."

# 或使用 .env 文件
cp .env.example .env
# 编辑 .env

# 2. 运行
python -m symphony WORKFLOW.md
```
