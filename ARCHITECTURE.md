# Symphony 架构

## 概述

Symphony 是一个智能 Agent 编排系统，使用 LLM 驱动的 Agent 自动化软件工程任务。它与 Linear 集成进行问题跟踪，并支持多个 LLM 供应商。

## 核心组件

### 1. 配置系统 (`symphony/config/`)

**用途**: 管理来自多个来源的配置，具有优先级排序。

**关键类**:
- `Config`: 主配置容器
- `LLMConfig`: LLM 供应商设置
- `TrackerConfig`: Linear 集成设置
- `WorkspaceConfig`: 工作空间管理设置

**配置优先级**:
1. WORKFLOW.md YAML 前置元数据
2. 环境变量
3. .env 文件
4. 默认值

### 2. LLM 客户端 (`symphony/llm/`)

**用途**: 支持多个后端的供应商无关 LLM 接口。

**关键类**:
- `LLMClient`: 所有供应商的统一接口
- `Message`: 消息格式标准化
- `LLMResponse`: 规范化响应处理

**支持的供应商**:
- OpenAI (GPT-4, GPT-3.5)
- Anthropic (Claude 3)
- DeepSeek
- Google Gemini
- Azure OpenAI

### 3. 编排器 (`symphony/orchestrator/`)

**用途**: 管理 Agent 生命周期的中央调度器。

**关键类**:
- `Orchestrator`: 主编排循环
- `OrchestratorState`: 状态持久化
- `RunningEntry`, `RetryEntry`: 状态跟踪

**职责**:
- 轮询 Linear 获取新问题
- 管理并发 Agent 槽位
- 使用指数退避处理重试
- 状态协调

### 4. Agent (`symphony/agents/`)

**用途**: 使用支持工具的 LLM 执行任务。

**关键类**:
- `SymphonyAgent`: 主 Agent 实现
- `AgentSession`: 会话状态管理

**工具**:
- `read_file` / `write_file`: 文件操作
- `execute_command`: Shell 命令执行
- `linear_graphql`: Linear API 查询
- `add_comment`: 向问题添加评论

### 5. 跟踪器 (`symphony/trackers/`)

**用途**: 与问题跟踪系统接口。

**关键类**:
- `LinearTracker`: Linear GraphQL API 客户端
- `Issue`: 规范化问题表示

### 6. 工作空间管理器 (`symphony/workspace/`)

**用途**: 管理 Agent 工作目录。

**关键类**:
- `WorkspaceManager`: 目录生命周期管理
- `WorkspaceContext`: 路径安全验证

**钩子**:
- `after_create`: 初始化工作空间
- `before_run`: 准备执行
- `after_run`: 执行后清理
- `before_remove`: 最终清理

### 7. 仪表板 (`symphony/dashboard/`)

**用途**: 实时监控的终端 UI。

**关键类**:
- `Dashboard`: 基于 Rich 的终端界面

**特性**:
- 运行中的 Agent 列表
- LLM Token 使用统计
- 重试队列状态
- 系统健康状态

## 安全考虑

### 路径安全
- 所有文件操作使用 `_resolve_path()` 确保路径停留在工作空间内
- 防止符号链接遍历
- 验证相对路径解析

### API 密钥处理
- 密钥从环境变量或 .env 文件加载
- 密钥永远不会在日志或错误消息中暴露
- 支持通过环境进行密钥轮换

### 钩子执行
- 钩子以可配置的超时时间运行
- Shell 命令在工作空间内被沙箱化
- 输出被捕获并记录

## 可扩展性

### 并发 Agent
- 可配置的最大并发 Agent 数
- 基于槽位的分配
- 优雅地处理资源限制

### 状态持久化
- 内存中的状态，可选持久化
- 重启时状态协调
- 原子状态转换

### 重试策略
- 失败时指数退避
- 单独队列用于连续重试
- 可配置的重试限制

## 扩展点

### 添加新的 LLM 供应商
1. 在 `symphony/llm/providers/` 中创建适配器类
2. 实现 `chat_completion()` 方法
3. 在 `LLMClient._get_provider_client()` 中注册

### 添加新工具
1. 在 `symphony/agents/tools/` 中定义工具函数
2. 添加到 `__init__.py` 中的 `__all__`
3. 在 Agent 的工具注册表中注册
4. 更新系统提示词以描述工具

### 自定义跟踪器
1. 实现 `BaseTracker` 接口
2. 覆盖 `poll()`, `claim()`, `complete()`
3. 在编排器工厂中注册

## 配置示例

### 最小配置
```yaml
symphony:
  version: "1.0"
  settings:
    llm:
      provider: openai
      model: gpt-4
    tracker:
      kind: linear
      project_slug: my-team
```

### 高级配置
```yaml
symphony:
  version: "1.0"
  settings:
    llm:
      provider: anthropic
      model: claude-3-opus-20240229
      temperature: 0.5
      max_tokens: 8192
    agent:
      max_turns: 50
      include_patterns: ["**/*.py", "**/*.ts"]
      exclude_patterns: ["**/tests/**"]
    tracker:
      kind: linear
      project_slug: my-team
      assignee: user-uuid
      active_states: ["Backlog", "Todo"]
    workspace:
      root: /var/symphony/workspaces
      max_concurrent_agents: 10
    hooks:
      after_create: |
        git clone $REPO_URL $SYMPHONY_WORKSPACE
```

## 开发

### 运行测试
```bash
make test          # 运行所有测试
make test-cov      # 带覆盖率运行
make lint          # 运行代码检查
make format        # 格式化代码
```

### Docker 开发
```bash
make docker-build  # 构建镜像
make docker-run    # 运行容器
make docker-logs   # 查看日志
```

### 本地开发
```bash
make install-dev   # 安装开发依赖
symphony init      # 初始化配置
symphony doctor    # 检查环境
symphony run       # 启动编排器
```
